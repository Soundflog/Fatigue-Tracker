## 3. Недостатки (по приоритету)

### 3.1 Критично: некорректная схема split в персонализации (одноклассовый test)

- **Проблема:** в персонализации используется последовательный разрез `[:70%] / [70%:]` внутри одного субъекта без стратификации по классу.
- **Где видно:** `diplom.ipynb`, code cell 24 (`X_subj_train = X_subject[:n_train]`, `X_subj_test = X_subject[n_train:]`), output с `UndefinedMetricWarning` и `ROC-AUC: nan`.
- **Риск:** метрики персонализации статистически некорректны; эффект fine-tuning нельзя интерпретировать как достоверный.
- **Что исправить:** заменить на `StratifiedShuffleSplit` (или эквивалент) по `y_subject`; добавить проверку наличия обоих классов в train/test до старта обучения и отчёта.

### 3.2 Критично: LOSO проведён на 5 случайных фолдах из 27

- **Проблема:** функция LOSO ограничивает число фолдов параметром `n_folds=5` и случайной выборкой субъектов.
- **Где видно:** `diplom.ipynb`, code cell 22 (`loso_evaluation(..., n_folds=5)` и блок `np.random.choice`).
- **Риск:** итоговая оценка обобщающей способности смещена и нестабильна; выводы о generalization переоценены.
- **Что исправить:** запускать LOSO на всех `27` субъектах по умолчанию; демо-режим фиксировать отдельным флагом и явно маркировать как неполную оценку.

### 3.3 Критично: валидационный набор слишком мал и смещён

- **Проблема:** `Val=476` окон (`1.9%`) и только 2 субъекта, оба из `zenodo`; доля positives `60.7%` существенно выше train/test.
- **Где видно:** `diplom.ipynb`, output cell 10 (`Train/Val/Test`), фактический состав вал-набора по субъектам.
- **Риск:** early stopping и выбор лучшей эпохи неустойчивы; высока вероятность случайного выбора модели.
- **Что исправить:** пересобрать split со стратификацией по субъектам/доменам; зафиксировать минимальные квоты на число субъектов и окон в val.

### 3.4 Высокий: несогласованность критериев early stopping и checkpoint

- **Проблема:** остановка обучения основана на `val_loss`, а сохранение лучшей модели идёт по `val_f1`.
- **Где видно:** `diplom.ipynb`, code cell 16 (`if val_f1 > best_val_f1: save`, но `if early_stopping(val_loss, model): break`).
- **Риск:** тренировка может остановиться до достижения лучшего F1, либо сохранить не ту модель, что оптимизируется при остановке.
- **Что исправить:** использовать единый критерий (например, `val_f1` для checkpoint и early stopping) либо явно задокументировать multi-objective логику с приоритетом.

### 3.5 Высокий: широкие `except` маскируют проблемы в расчёте метрик

- **Проблема:** метрики в нескольких местах обёрнуты в `try/except` с fallback значениями (`0.5`), без фиксации причины.
- **Где видно:** `diplom.ipynb`, code cell 14 и 19 (блоки расчёта ROC-AUC и PR-AUC).
- **Риск:** silent failure; отчёт может содержать численно валидные, но методологически ложные значения.
- **Что исправить:** ловить конкретные исключения (`ValueError`), записывать статус метрики (`ok/undefined/error`) и причину, не подменять автоматически на `0.5`.

### 3.6 Высокий: отсутствует полноценное фиксирование seed

- **Проблема:** seed фиксируется только для части numpy-операций; нет единой инициализации `random`, `numpy`, `torch`, `torch.cuda`.
- **Где видно:** `diplom.ipynb`, split-функции и случайный выбор субъекта в персонализации.
- **Риск:** результаты между запусками различаются, сравнимость экспериментов ухудшается.
- **Что исправить:** ввести единую функцию `set_seed(seed)` и вызывать до всех шагов split/train/eval.

### 3.7 Высокий: фиксированный порог `0.5` без подбора на validation

- **Проблема:** бинаризация вероятностей всегда через `>=0.5`.
- **Где видно:** `diplom.ipynb`, code cell 14 и 19.
- **Риск:** недоиспользование качества модели на целевой метрике; текущий отчёт уже показывает дисбаланс recall между классами.
- **Что исправить:** подбирать threshold на validation (максимум `F1-macro` или баланс `recall_0/recall_1`) и переносить этот порог на test.

### 3.8 Высокий: доменный дисбаланс не компенсирован методологически

- **Проблема:** объём `4tu` примерно в 3.14 раза больше `zenodo`, но нет явной доменной балансировки/взвешивания/анализа сдвига.
- **Где видно:** `diplom.ipynb`, output cell 5 (`18862 vs 6006`).
- **Риск:** модель может учить доменный сигнал вместо физиологического признака усталости; переносимость между источниками не подтверждена.
- **Что исправить:** добавить доменно-стратифицированные протоколы и отчёт per-domain; при необходимости использовать reweighting или balanced sampling.

### 3.9 Средний: высокая межсубъектная дисперсия LOSO не разобрана

- **Проблема:** зафиксирован широкий разброс `F1` (от `0.3631` до `0.9048`, `std=0.1971`), но причинный анализ не выполнен.
- **Где видно:** `results/loso_results.csv` и outputs cell 22.
- **Риск:** средняя метрика скрывает провалы на отдельных спортсменах; риск регрессий в персонализации.
- **Что исправить:** добавить обязательный пер-субъектный error analysis (размер выборки, баланс классов, домен, сложные субъекты).

### 3.10 Средний: несоответствие техдокументации и фактического эксперимента

- **Проблема:** в техдоке указаны LOSO-сценарии на `37/38` фолдов, фактически в ноутбуке используется `27` субъектов и демо на `5` фолдах.
- **Где видно:** `docs/ТЕХНИЧЕСКАЯ_ДОКУМЕНТАЦИЯ.md` (разделы по LOSO) против `diplom.ipynb`, cell 22.
- **Риск:** расхождение между отчётной и воспроизводимой частью диплома.
- **Что исправить:** синхронизировать техдок с фактической версией датасета и протокола, указать дату/ревизию эксперимента.

### 3.11 Средний: рассинхрон описания входа модели (`30` vs `100`)

- **Проблема:** в docstring модели встречается `Вход: (batch, 30, 6)`, при фактическом формате данных `100` точек.
- **Где видно:** `diplom.ipynb`, code cell 12 и выводы загрузки данных.
- **Риск:** ошибка в коммуникации требований, неправильная репликация модели.
- **Что исправить:** унифицировать описание во всех markdown/docstring/отчётах на `100` точек.

### 3.12 Средний: ноутбук монолитный и дублирует логику проектных модулей

- **Проблема:** в ноутбуке заново определяются dataset/model/train/eval, хотя аналогичная логика уже есть в `afc/models_cnn.py` и `scripts/train_deep.py`.
- **Где видно:** `diplom.ipynb`, code cell 12-24; проектные модули `afc/models_cnn.py`, `scripts/train_deep.py`.
- **Риск:** расхождение поведения между "ноутбук-версией" и "скрипт-версией"; сложнее сопровождать.
- **Что исправить:** оставить в ноутбуке orchestration/визуализацию, а тренировочную и метрик-логику импортировать из модулей.

## 4. План улучшения и развития

### Этап A (1-3 дня): воспроизводимость и корректность метрик

- **Срок:** 1-3 рабочих дня.
- **Действия:**
  - Ввести единый `set_seed(seed)` для `random/numpy/torch`.
  - Убрать широкие `except`; добавить явные статусы метрик (`ok`, `undefined`, `error`).
  - Согласовать критерий `early stopping` и `best checkpoint`.
  - Зафиксировать версию зависимостей (особенно `scikit-learn`) в отчёте эксперимента.
- **Артефакты:**
  - Обновлённый ноутбук и/или модуль метрик.
  - Лог запуска с seed и версиями библиотек.
  - Таблица метрик с флагом валидности.
- **Критерий готовности:**
  - Ни одна метрика не подменяется "тихо".
  - Любой `nan/undefined` явно объяснён в отчёте.
  - Два повторных запуска с одинаковым seed дают совпадающие результаты (с малой численной погрешностью).

### Этап B (3-7 дней): исправление протоколов валидации

- **Срок:** 3-7 рабочих дней.
- **Действия:**
  - Пересобрать split с доменно-субъектной стратификацией.
  - Увеличить и стабилизировать `val` (по числу субъектов и окон).
  - Запускать полный LOSO на всех субъектах.
  - Добавить две сводки LOSO: unweighted и weighted (по `n_samples`).
- **Артефакты:**
  - Новый CSV полного LOSO (все субъекты).
  - Отдельный отчёт по train/val/test составу (subjects/domains/classes).
  - Репродуцируемый скрипт протокола.
- **Критерий готовности:**
  - В каждом split присутствуют оба класса.
  - Нет утечки субъектов между train/val/test.
  - LOSO покрывает 100% субъектов.

### Этап C (1-2 недели): улучшение качества модели

- **Срок:** 1-2 недели.
- **Действия:**
  - Подобрать threshold по validation.
  - Добавить калибровку вероятностей (Platt/Isotonic с валидацией).
  - Провести анализ ошибок по доменам, субъектам, классам.
  - Запустить базовый hyperparameter search (LR, dropout, weight decay, batch size).
- **Артефакты:**
  - Таблица сравнений baseline vs tuned.
  - Графики calibration/ROC/PR и per-domain confusion matrices.
  - Конфигурации лучших запусков.
- **Критерий готовности:**
  - Улучшение целевой метрики на фиксированном протоколе.
  - Снижение асимметрии recall классов.
  - Стабильность метрик на повторных запусках.

### Этап D (2-6 недель): развитие до production-ready уровня диплома

- **Срок:** 2-6 недель.
- **Действия:**
  - Включить cross-dataset протоколы (train Zenodo -> test 4TU и наоборот).
  - Перенести ключевую логику из ноутбука в скрипты/модули (`split`, `metrics`, `experiment_config`).
  - Ввести единый формат `experiment report` и версионирование результатов.
  - Подготовить стабильный pipeline оценки (`eval_report` + шаблон отчёта).
- **Артефакты:**
  - Модульные утилиты для split/metrics/config.
  - Скрипт воспроизведения полного цикла эксперимента.
  - Консолидированный отчёт для диплома.
- **Критерий готовности:**
  - Один CLI-запуск воспроизводит эксперимент и формирует полный комплект артефактов.
  - Документация и фактические результаты синхронизированы.

## 5. Критерии приёмки и проверки

1. **Проверка split-корректности.**
   - Нет пересечения `sid` между train/val/test.
   - В каждом split есть оба класса.

2. **Проверка edge-cases метрик.**
   - Для одноклассовых наборов метрика помечается `undefined` с причиной.
   - Нет автоматической подмены на константы без логирования.

3. **Проверка воспроизводимости.**
   - Три запуска с фиксированным seed дают близкие метрики (допуск задаётся в протоколе, например `|delta F1| <= 0.01`).

4. **Проверка LOSO-покрытия.**
   - Количество фолдов равно количеству субъектов.
   - Сохранён пер-субъектный CSV с `n_samples`, `f1_macro`, `roc_auc`, `pr_auc`, `balanced_acc`.

5. **Проверка персонализации.**
   - Используется стратифицированный split внутри субъекта.
   - На test персонализации отсутствуют `nan` по ключевым метрикам (или явно объяснены как `undefined`).

6. **Проверка отчётности.**
   - В отчёте присутствуют baseline, mean/std, weighted/unweighted агрегаты.
   - Есть интервальные оценки неопределённости (например, bootstrap CI).

## 6. Риски и зависимости

### 6.1 Ключевые риски

1. **Вычислительный риск:** полный LOSO на CPU существенно дольше демо-режима (в ноутбуке 5 фолдов уже занимают порядка 18 минут).
2. **Риск данных:** порядок шагов внутри субъекта влияет на персонализацию при наивном sequential split.
3. **Риск совместимости:** поведение `roc_auc_score` в одноклассовых случаях зависит от версии `scikit-learn`.
4. **Методологический риск:** без cross-dataset оценки остаётся неопределённой переносимость модели между источниками.

### 6.2 Зависимости для выполнения roadmap

1. Актуальный и зафиксированный набор зависимостей (`requirements` с версиями).
2. Доступ к полному датасету `composite_strides.npz` и артефактам `results`.
3. Единый конфиг эксперимента (пути, seed, протоколы, метрики, порог).
4. Базовые ресурсы для длительных прогонов LOSO/cross-dataset.

### 6.3 Явные допущения и выбранные дефолты

1. Анализируется `notebooks/diplom.ipynb` (запрос `diplmon.ipynb` интерпретирован как опечатка).
2. Язык отчёта: русский.
3. Формат: Markdown, UTF-8.
4. Изменений публичных API в рамках этого запроса не вносится.
5. В roadmap допустимы будущие additions: `split`, `metrics`, `experiment_config` без ломки внешнего контракта текущих скриптов.
